Geese Spotter Game

C++ | Algorithmic Design & Memory Management
Back to Portfolio

Project Overview

Geese Spotter is a terminal-based strategy game heavily inspired by the classic logic puzzle Minesweeper. The core objective of this project was not just to create a playable game, but to build a robust foundation in fundamental computer science principles using C++. By interacting directly with the command line, I focused entirely on underlying architectural efficiency rather than graphical overhead, demonstrating a strong grasp of raw memory manipulation and algorithmic logic.

Architecture & Design

A major design constraint of this project was representing a two-dimensional game board in memory efficiently. Instead of using a standard library container like std::vector or a 2D array, I implemented the board using a dynamically allocated 1D array accessed via raw pointers. This ensures contiguous memory allocation, which is highly cache-friendly and faster for the CPU to process.

To access a specific (X, Y) coordinate on the board, I utilized pointer arithmetic to calculate the exact index in the 1D array: Index = Y * Width + X. This mathematical mapping is a crucial concept in low-level graphics and embedded system displays.

To further optimize memory, I used bitwise operations to track the state of each tile. A single integer per tile was used to store multiple Boolean states (e.g., Does it contain a goose? Is it revealed? Is it flagged?) by setting and checking specific bits using bitwise AND (&), OR (|), and XOR (^) operators. This drastically reduced the memory footprint of the game board.

Challenges & Solutions

Challenge: Implementing the "Zero-Cascade" Reveal Algorithm. In Minesweeper, clicking a tile with zero neighboring hazards automatically reveals all adjacent safe tiles, which can in turn reveal more tiles. Implementing this cascading effect efficiently without crashing the program via stack overflow was a significant algorithmic challenge.

Solution: Recursive Depth-First Search (DFS). I implemented a Depth-First Search algorithm to handle the cascade. When a zero-hazard tile is revealed, the function recursively calls itself on all 8 surrounding neighbors. To prevent infinite loops and stack overflows, I added strict base cases: the algorithm halts if a neighbor is out of bounds, already revealed, or contains a goose. This guaranteed O(N) time complexity where N is the number of safe tiles connected to the origin.

Challenge: Strict Memory Leak Prevention. Because the board was dynamically allocated using the new keyword, restarting the game or exiting the program required explicit memory cleanup.

Solution: I developed robust teardown functions ensuring the delete[] operator was reliably called under all exit conditions. I utilized Valgrind/GDB during the debugging process to verify that zero memory leaks occurred over thousands of game iterations.
