Why Async FIFO Verification Is Hard

An Asynchronous FIFO is one of the most common — and most dangerous — building blocks in chip design. Its purpose is to safely transfer data between two independent clock domains (a write clock and a read clock) that are completely unsynchronized with each other.

The Problem

Two clocks running at different frequencies create metastability — flip-flops can get stuck in undefined states when signals cross clock domains at the wrong moment. A standard FIFO would corrupt data or miss transitions entirely.

The DUT Solution

The async FIFO uses Gray code pointers — only 1 bit changes per increment — and double-flop synchronizers to safely cross the clock boundary. The testbench must exercise both full and empty conditions across both domains.

Verifying this properly requires driving write-side stimulus on wclk (200ns period) and read-side stimulus on rclk (300ns period) simultaneously, checking that every byte written eventually comes out the other side uncorrupted — regardless of timing.

UVM organizes a testbench into a strict hierarchy of reusable components. Each layer has exactly one job. The diagram below shows the complete structure of this project:

Key Design Challenges & Solutions

1. Bridging Non-Blocking and Blocking TLM Ports

Problem

Monitors call ap.write() which is non-blocking — fire and forget. The model and scoreboard need blocking get() calls to wait for data. You can't directly connect the two.

Solution

Three uvm_tlm_analysis_fifo buffers (agt_mdl_fifo, agt_scb_fifo, mdl_scb_fifo) sit between them. The FIFO accepts non-blocking writes and provides blocking reads — bridging the two port types cleanly.

// env connect_phase — the three FIFO bridges i_agt.ap.connect(agt_mdl_fifo.analysis_export); // monitor → FIFO mdl.port.connect(agt_mdl_fifo.blocking_get_export); // FIFO → model mdl.ap.connect(mdl_scb_fifo.analysis_export); // model → FIFO scb.exp_port.connect(mdl_scb_fifo.blocking_get_export); // FIFO → scoreboard o_agt.ap.connect(agt_scb_fifo.analysis_export); // read monitor → FIFO scb.act_port.connect(agt_scb_fifo.blocking_get_export); // FIFO → scoreboard

2. Backpressure — Handling a Full FIFO

Problem

If the driver blindly asserts winc every clock cycle, it will try to write into a full FIFO — data is lost and the test will fail with compare mismatches in the scoreboard.

Solution

The driver checks wfull on every clock edge before driving. If full, it deasserts winc and waits another clock — a backpressure loop. It only proceeds once the FIFO signals it has space.

// asyncf_driver::drive_one_pkt — backpressure loop while(1) begin if (up_if.wfull) begin up_if.winc <= 1'b0; // FIFO full — deassert and wait @(posedge up_if.wclk); end else begin up_if.winc <= 1'b1; // space available — drive data up_if.wdata <= data_q[i]; break; end end

3. Safe Transaction Passing in the Model

Problem

If the model calls ap.write(tr) with the same object handle every loop iteration, the next port.get() overwrites the memory that the scoreboard still holds a pointer to — corrupting the expected queue.

Solution

The model allocates a fresh new_tr = new() object each iteration and copies into it with new_tr.copy(tr). The scoreboard receives a pointer to memory that the model no longer touches.

// asyncf_model::main_phase — safe copy pattern while(1) begin port.get(tr); // receive write transaction new_tr = new("new_tr"); // allocate FRESH memory every iteration new_tr.copy(tr); // copy fields into new object ap.write(new_tr); // send safe pointer to scoreboard end

4. Coordinating Two Clock Domains

Problem

The write driver runs on wclk (200ns period) and the read driver runs on rclk (300ns period). They need to operate simultaneously without one blocking the other.

Solution

The virtual sequencer holds handles to both sequencers. The virtual sequence uses uvm_do_on to target each sequencer independently — write operations go to m_up_seqr, reads go to m_down_seqr.

// asyncf_case0_sequence::body() — coordinating both domains `uvm_declare_p_sequencer(asyncf_virtual_sequencer) repeat(7) `uvm_do_on(up_seq, p_sequencer.m_up_seqr) // writes on wclk repeat(7) `uvm_do_on(down_seq, p_sequencer.m_down_seqr) // reads on rclk

Common Interview Questions

Q: What is UVM and why is it used?
UVM is a standardized SystemVerilog library for building reusable, scalable verification environments. It provides a class hierarchy, phase mechanism, factory, and TLM communication infrastructure. It's used because manually written testbenches don't scale — UVM lets teams reuse components across projects and achieve constrained-random coverage-driven verification.
Q: What is the difference between uvm_object and uvm_component?
uvm_object is the base for data — transactions and sequences. It has no parent and no phases. It floats in memory and gets created/destroyed as needed. uvm_component is the base for structural pieces — driver, monitor, agent, env. It has a fixed position in the hierarchy tree, a parent pointer, and participates in UVM phases (build, connect, run). Constructor signature reveals the difference: objects take only a name, components take name + parent.
Q: What does the scoreboard do and how does it receive data?
The scoreboard compares expected vs actual transactions. It has two blocking_get_ports — exp_port fed by the reference model, act_port fed by the read-side monitor. Two fork threads run in parallel: one pushes expected transactions into a queue, the other pops them and calls compare() against the actual DUT output. A mismatch triggers uvm_error which increments the global error counter and eventually causes TEST CASE FAILED.
Q: Why use a virtual sequencer?
The virtual sequencer coordinates multiple real sequencers from one place. Without it, the test case would need to directly call start() on both the write and read sequencers separately. The virtual sequencer holds handles to both (m_up_seqr and m_down_seqr) and the virtual sequence uses uvm_do_on to direct stimulus to the correct one — keeping the test clean and domain-agnostic.
Q: What is uvm_config_db used for?
It's a global key-value store shared across the entire testbench. top_tb.sv uses set() to store the virtual interface handles (actual hardware connections) keyed to specific component paths. The driver and monitor use get() to retrieve those handles in their build_phase. This decouples the hardware world (module-space) from the software world (class-space) without requiring direct hierarchical references.
Q: Why is raise/drop objection needed?
UVM ends the run phase as soon as there are zero objections. Without raise_objection, simulation ends immediately when main_phase starts — before any stimulus is sent. The top-level virtual sequence calls raise_objection in pre_body to keep simulation alive, and drop_objection in post_body after all stimulus is complete. This is the UVM mechanism for controlling simulation end time.
