Chess Clock FSM

Project Overview

I designed a fully synthesizable two-player chess clock in SystemVerilog, implemented as a finite state machine (FSM) with edge-detected turn switching, a 1 Hz tick generator for real-time countdown, and asynchronous reset. The module cleanly separates next-state logic (combinational) from state storage and datapath updates (sequential), following the standard two-process FSM pattern used in production RTL.

The design targets an FPGA running at 100 MHz and correctly converts that fast clock into a human-readable seconds countdown — solving the fundamental problem of bridging silicon-speed digital logic with real-world time.

What Problem Are We Solving?

A chess clock is deceptively simple: two countdown timers, one button to switch turns. But implementing it in hardware reveals several non-obvious challenges that mirror real-world RTL design problems.

Problem 1: Button Bounce

A physical button press lasts thousands of clock cycles at 100 MHz. Without edge detection, the FSM would oscillate between PLAYER1 and PLAYER2 millions of times per press — making the clock unusable.

Solution: Rising-Edge Detector

A single flip-flop delays the button signal by one cycle. ANDing the current value with the inverted previous value produces exactly one pulse per press — clean, deterministic switching.

Problem 2: Clock Speed vs Human Time

The FPGA clock ticks 100 million times per second. Decrementing the timer every cycle would drain a 60,000-second timer in 0.6 ms — invisible to a human player.

Solution: 1 Hz Tick Generator

A 27-bit counter counts from 0 to 99,999,999 and wraps, producing a single-cycle tick pulse every second. The timer only decrements when tick is asserted.

Problem 3: Multiple Drivers & Latches

Driving the same signal from both an always_ff block and a continuous assign creates synthesis errors. Incomplete case coverage creates unintended latches — a common beginner bug.

Solution: Strict Two-Process Separation

One always_comb for next-state logic, one always_ff for all sequential updates. Every signal is driven from exactly one block, and defaults at the top of always_comb prevent latches.

The clock has four states, encoded in a 2-bit register. The design is intentionally simple — each state has exactly one responsibility, and transitions are guarded by clear conditions.

Edge Detection — How It Works

The edge detector is one of the most useful small circuits in digital design. It converts a level (button held down) into a single-cycle pulse (one clean event).

player_switch ──┬──────────────────────────────────► AND ──► switch_edge │ ↑ └──► [D Flip-Flop] ──► switch_prev ──► NOT ──┘ // Hardware: 1 flip-flop + 1 inverter + 1 AND gate // Result: switch_edge = player_switch & ~switch_prev // HIGH for exactly ONE clock cycle on rising edge

The timing table below shows exactly when switch_edge fires:

switch_edge 00100000

Cycle 3 is the only cycle where player_switch=1 AND switch_prev=0 — one clean pulse regardless of how long the button is held.

1 Hz Tick Generator

The tick generator is a modulo counter that converts the 100 MHz FPGA clock into a once-per-second pulse. It counts from 0 to 99,999,999 and wraps, asserting tick for exactly one cycle at the rollover point.

// Counter: 27-bit register counts 0 → 99,999,999 → 0 → ... always_comb begin if (tick_counter == CLK_FREQ_HZ - 1) tick_counter_next = '0; // wrap to zero else tick_counter_next = tick_counter + 1; end // tick is HIGH for exactly 1 cycle every 100M cycles (= 1 second) assign tick = (tick_counter == CLK_FREQ_HZ - 1);

Why 27 bits? $clog2(100,000,000) = 27. You need 27 bits to represent numbers up to 100 million. The parameter CLK_FREQ_HZ makes this portable — change it to 50,000,000 for a 50 MHz board and the tick generator automatically adjusts.

Module Architecture

The design uses the two-process FSM pattern — one combinational block for next-state decisions, one sequential block for all register updates. This is the standard pattern taught in ECE 327 and used in industry.

chess_clock module │ ├── Edge Detector (1 FF + AND + NOT) │ └── player_switch → switch_prev → switch_edge │ ├── Tick Generator (27-bit counter) │ └── tick_counter → tick_counter_next → tick │ ├── always_comb (BLOCK 1: Next-State Logic) │ └── state + switch_edge + p1_time + p2_time → state_next │ └── always_ff (BLOCK 2: State + Datapath) ├── state ← state_next ├── p1_time ← p1_time - 1 (when PLAYER1_TURN & tick) ├── p2_time ← p2_time - 1 (when PLAYER2_TURN & tick) └── gameover ← 1 (when GAMEOVER)

Key Code Sections

Next-State Logic (Combinational)

This block computes the next state from the current state and inputs. The default assignment state_next = state at the top prevents latches — if no condition triggers, the state holds.

always_comb begin state_next = state; // default: hold case (state) IDLE: begin if (switch_edge) state_next = PLAYER1_TURN; end PLAYER1_TURN: begin if (player1_time == 0) state_next = GAMEOVER; // time ran out → highest priority else if (switch_edge) state_next = PLAYER2_TURN; // button press → switch turns end PLAYER2_TURN: begin if (player2_time == 0) state_next = GAMEOVER; else if (switch_edge) state_next = PLAYER1_TURN; end GAMEOVER: state_next = GAMEOVER; // latched — only rst_n escapes endcase end

Datapath Updates (Sequential)

This block updates all registers on the clock edge. The key insight: in always_ff, if you don't assign a register, it holds its value. So during PLAYER1_TURN, player2_time is not assigned and stays frozen — which is exactly correct.

always_ff @(posedge clk or negedge rst_n) begin if (!rst_n) begin state <= IDLE; player1_time <= TIMER[15:0]; player2_time <= TIMER[15:0]; gameover <= 1'b0; end else begin state <= state_next; case (state) IDLE: begin player1_time <= TIMER[15:0]; // hold at starting time player2_time <= TIMER[15:0]; gameover <= 1'b0; end PLAYER1_TURN: begin if (tick && player1_time > 0) player1_time <= player1_time - 1; // player2_time not assigned → holds value automatically end PLAYER2_TURN: begin if (tick && player2_time > 0) player2_time <= player2_time - 1; end GAMEOVER: gameover <= 1'b1; endcase end end

Key Concepts & Takeaways

Two-Process FSM Pattern
Separate next-state logic (combinational) from state storage (sequential). This eliminates common bugs: latches from incomplete combinational coverage, multiple drivers, and mixing blocking/non-blocking assignments. It's the standard pattern in production RTL.
Edge Detection vs Level Detection
Level detection (checking if a signal IS high) causes repeated triggers every clock cycle. Edge detection (checking if a signal BECAME high) produces exactly one pulse per event. The pattern — one flip-flop to delay, AND with inverted previous — is reusable everywhere: buttons, interrupt lines, handshake signals.
Clock Division Without a PLL
A simple modulo counter converts any clock frequency to any slower frequency. Count to N-1, then wrap. Assert a tick pulse at the wrap point. This avoids needing a PLL or clock management tile for simple timing applications.
Implicit Hold in always_ff
In sequential blocks, an unassigned register retains its value — flip-flops hold by default. This is the opposite of always_comb where every signal must be assigned in every path (or latches are inferred). Understanding this distinction prevents one of the most common RTL bugs.
Parameterization for Portability
TIMER and CLK_FREQ_HZ as parameters mean the module works on any board at any clock frequency without changing a single line of RTL. This is the same approach used in industrial IP cores that ship to hundreds of different customers.
