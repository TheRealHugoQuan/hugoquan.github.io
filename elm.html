Project Overview

This project implements a full Extreme Learning Machine (ELM) neural network accelerator on an FPGA, targeting real-time classification and prediction tasks in embedded medical applications. Rather than offloading all computation to a CPU, the core matrix operations are executed directly in programmable logic using a hardware/software co-design methodology on a Xilinx Zynq SoC.

The system supports two deployment modes — non-pretrained (weights generated randomly at runtime and trained on-device) and pretrained (weights computed offline via Python and loaded from a .mem file) — and exposes all hardware modules as AXI-interfaced IP cores for seamless ARM-FPGA communication.

What is an Extreme Learning Machine?

Standard neural networks (e.g. MLPs trained with backpropagation) require many iterative passes through training data to converge. ELM sidesteps this entirely. It is a Single Hidden Layer Feedforward Network (SLFN) where:

1

Input-to-hidden weights and biases are randomly initialized once and never updated. No gradient needed for this layer.

2

The hidden layer output matrix H is computed by passing all training samples through the random projection and activation function.

3

Output weights β are solved analytically in one shot using least-squares: β = H⁺ · T, where H⁺ is the Moore-Penrose pseudoinverse of H.

This makes ELM dramatically faster to train than backpropagation-based networks, and its fixed computation graph makes it highly amenable to hardware acceleration. The key challenge — and the main hardware contribution of this project — is computing the pseudoinverse efficiently on an FPGA.

y_j = Σ βᵢ · g(aᵢ · xⱼ + bᵢ),   j = 1, 2, ..., N
β = H⁺ · T   where   H⁺ = Hᵀ(HHᵀ)⁻¹

Stage 1 — Givens Rotation QR Decomposition (GRQRD)

The goal is to factorize the hidden layer output matrix H into an orthogonal matrix Q and an upper triangular matrix R such that H = QR. Givens rotation zeros out one sub-diagonal element at a time by applying a 2×2 rotation matrix parameterized by cosine (c) and sine (s) factors:

r₁(n) = √( |r₁(n−1)|² + |x₁(n)|² )   — update hypotenuse
c = r₁(n−1) / r₁(n)    s = x₁(n) / r₁(n)   — compute rotation factors
rₖ(n) = c · rₖ(n−1) + s · xₖ(n)   — apply rotation to remaining elements

The systolic array that implements this has two types of processing elements arranged in a triangle:

GG Unit — Boundary (Vectoring Mode)

Located on the diagonal. Receives the first element of each column, computes the Givens rotation angle d, and produces the rotation factors c and s which are broadcast rightward to the GR units in the same row.

GR Unit — Internal (Rotation Mode)

Located in the interior. Receives c and s from the GG unit and applies the rotation transformation to its element. Its accumulated value rₖ is passed downward to the next boundary unit in the same column.

Because each column of the systolic array has a different number of PE stages, output data arrives at different times. Shift registers are added at the output to re-align results into a coherent R matrix.

Stage 2 — Triangular Matrix Inversion (TMI)

After QR decomposition, only the upper triangular R matrix (size L×L) needs to be inverted. This is done via back-substitution — a much simpler and more hardware-efficient operation than general matrix inversion:

v_ii = 1 / u_ii   (diagonal: just take reciprocal)
v_ij = − ( Σ v_jk · u_ik ) / u_ii   (off-diagonal: back-substitute)

Stage 3 — Final Matrix Multiplication

The result R⁻¹ is multiplied with Qᵀ (the transpose of Q from the QR step) using a second systolic array for the matrix multiply:

H⁺ = R⁻¹ · Qᵀ     h_ij = Σ v_ki · q_kj

CORDIC for Hardware Trigonometry

Computing c = cos(θ) and s = sin(θ) for the Givens rotation would normally require floating-point hardware or expensive DSP operations. Instead, the CORDIC (Coordinate Rotation Digital Computer) algorithm was used — a purely iterative approach that uses only additions and bit-shifts.

// CORDIC Key Property No multipliers required for the rotation computation. The algorithm iteratively converges to the correct angle by rotating a vector through pre-computed fixed angles, using only adders and shift registers.

Implementation Details

3 iterations per clock cycle. Each GG and GR processing element performs 3 CORDIC iterations every clock, giving a good balance between precision and clock rate.

Fixed-Point Format

26-bit signed fixed-point: 1 sign bit + 13 integer bits + 12 fractional bits. All internal arithmetic in both GG and GR units operates at this precision.

Feedforward Network Hardware Design

Neuron Module

Each hardware neuron performs: multiply-accumulate of serial input with weight and bias data, followed by a non-linear activation function. The design supports two operational modes via a pretrained control signal:

pretrained = 1 (Offline Mode)

Weights and biases are pre-computed in Python offline and stored in a .mem binary file. At runtime, the Weight Memory (WM) and Bias Memory (BM) modules load these from external storage into internal block RAM before inference begins.

pretrained = 0 (Online Mode)

The ARM processor generates random weights and biases at runtime (using C's math.h) and streams them via DMA into the WM and BM modules. The network is trained on-device with live data — suitable for adaptive/online learning scenarios.

Activation Functions

Two activation functions were implemented in RTL, selectable at synthesis time via a Python-generated Tcl script:

// Sigmoid σ(x) = 1 / (1 + e⁻ˣ)

Involves division and exponential — both are expensive in hardware. A naive implementation would consume significant LUT and DSP resources and would be slow.

→ Solution: Symmetric LUT with PosFlag

Leverages the mathematical symmetry of Sigmoid: σ(−x) = 1 − σ(x). Only the positive half of the function is stored in a lookup table (SigMemory block RAM). A PosFlag register tracks the sign of the input, and a comparator reconstructs the negative half from the stored positive values — halving the required memory with no accuracy loss.

// ReLU f(x) = max(0, x)

Piecewise linear — zero for negative inputs, identity for positive inputs.

→ Implementation: Comparator + Multiplexer

Essentially free in hardware. A single comparator checks the sign bit of the input. A multiplexer selects between zero and the input value accordingly. No LUT storage or arithmetic required beyond what is already present in the data path.

Connection Layer — Serial/Parallel Conversion

To balance resource usage against throughput, the design adopts a layer-serial, layer-parallel architecture:

→

Between layers: data flows serially (one element per clock) to minimize wiring and fanout.

∥

Within a layer: all neurons compute in parallel simultaneously for maximum throughput.

The conversion between serial and parallel domains is handled by a finite state machine (FSM) with two states: IDLE (initializes control logic) and SEND (shifts out the parallel result serially to the next layer, one element per clock cycle).

Systolic Array Matrix Multiplication

Matrix multiplication is implemented using a triangular systolic array — a data-flow architecture where computation units are arranged in a grid and data streams rhythmically through adjacent cells each clock cycle.

For an input matrix A (m×n) and weight matrix B (n×k), the output C has dimensions m×k. The systolic flow proceeds as:

1

Initialize: Load weight matrix B into the systolic array PEs. These values remain fixed throughout the computation.

2

Input streaming: Split matrix A into m column-wise data streams of n elements each. These stream left-to-right through the array.

3

Compute: Each clock cycle, every PE multiplies its input by the locally stored weight, accumulates the result, and passes data to the next PE.

4

Output: Results drain from the array row-by-row into the output matrix C.

Because the ELM hidden layer matrix H is square (N×N in the simplified implementation), the systolic array dimensions are uniform, simplifying the control logic.

Software/Hardware Co-Design on Zynq

The Xilinx Zynq XC7Z020 integrates a dual-core ARM Cortex-A9 (the PS — Processing System) tightly coupled to FPGA fabric (the PL — Programmable Logic) on a single die. This project exploits both halves in a coordinated pipeline.

AXI Bus Interface

All hardware modules are wrapped as AXI-compatible IP cores. Two AXI variants are used:

AXI4-Stream (HP Port)

Used for high-throughput data movement — streaming the H matrix into the QR module, streaming results back out. Connected via DMA to the PS DDR memory. The ARM never touches individual bytes; it simply starts a DMA transfer and waits for a completion interrupt.

AXI4-Lite (GP Port)

Used for low-bandwidth register-level control — the ARM uses this to configure and start the DMA engine, set transfer lengths, and read status flags. Lightweight single-beat transfers only.

DMA Data Flow — QR Module Example

1

QR decomposition RTL is synthesized, placed, and stored to SDRAM. The QR module is wrapped with AXI-Stream ports.

2

ARM prepares the input H matrix in DDR memory, then configures the DMA controller (via AXI-Lite GP port) with the source address and transfer length.

3

DMA engine streams data from DDR → QR module via the high-performance HP AXI slave port. The ARM core is free to do other work during this transfer.

4

QR module processes the stream and outputs R and Q matrices via AXI-Stream. DMA writes results back to DDR. ARM receives a completion interrupt, reads results from DDR, and proceeds to compute β = R⁻¹·Qᵀ·T.

Key Challenges & Solutions

Challenge 1: Fixed-Point Precision in QR Decomposition

Givens rotation accumulates small rounding errors across iterations, which can cause numerical instability — especially when processing ill-conditioned matrices. Standard 16-bit fixed-point formats lose too much fractional precision during CORDIC iterations.

Solution

A custom 26-bit signed fixed-point format was designed: 1 sign bit + 13 integer bits + 12 fractional bits. The 12 fractional bits provide sufficient precision for CORDIC convergence across 3 iterations per clock, while 13 integer bits prevent overflow for the expected range of ELM weights.

Challenge 2: Output Timing Skew in the Systolic Array

Different columns of the triangular systolic array have different depths (number of PE stages). This causes output values to arrive at different clock cycles, making it impossible to directly read out a coherent matrix.

Solution

Programmable shift register chains are inserted at each column output to delay earlier-finishing columns, re-aligning all output data to the same clock boundary. The delay for each column is statically determined at synthesis time based on the array dimensions.

Challenge 3: Sigmoid Hardware Complexity

A full Sigmoid implementation requires an exponential and a division — two of the most resource-intensive operations in digital hardware.

Solution

Replaced with a half-table LUT exploiting Sigmoid symmetry. Only positive-domain values are stored in block RAM. A sign-check logic block reconstructs the full function range using the identity σ(−x) = 1 − σ(x), with a comparator selecting the right output path.

Verilog Deep Dive — Module Walkthroughs

The following sections walk through the two core RTL modules written for this project: the neuron module (the fundamental compute unit of the ELM feedforward network) and the AXI4-Lite wrapper (the complete ARM↔FPGA register interface). Each block of code is annotated with what it does and why the design choices were made.

Module 1 — neuron.v

The neuron module is the fundamental compute unit. Every hidden-layer and output-layer neuron in the ELM is an instantiation of this single parameterized module. It performs: weight memory read, multiply-accumulate across all inputs, saturating bias addition, and activation function application — all pipelined across a stream of serial input data.

Parameters & Compile-Time Configuration

// module declaration
module neuron #(
    parameter layerNo     = 0,       // which layer this instance belongs to
    parameter neuronNo    = 0,       // which neuron index within that layer
    parameter numWeight   = 784,     // 28×28 = MNIST input size
    parameter dataWidth   = 16,     // 16-bit fixed-point data paths
    parameter sigmoidSize = 5,      // top 5 bits of sum → Sigmoid LUT address
    parameter weightIntWidth = 1,   // 1 integer bit + 15 fractional bits (Q1.15)
    parameter actType     = "relu", // selects activation at synthesis time
    parameter biasFile    = "",     // path to .mem file for pretrained bias
    parameter weightFile  = ""      // path to .mem file for pretrained weights
)

All parameters are resolved at synthesis time — they are baked into the netlist. There is no runtime overhead. layerNo and neuronNo act as hardware identity tags: every neuron in the design sees the same weight-loading bus, but only the one whose layerNo and neuronNo match the incoming config signals will actually write.

// Why numWeight=784? 784 = 28 × 28 pixels — the flat input vector size for MNIST handwritten digit classification. The address width for the weight memory is automatically derived as $clog2(784) = 10 bits, giving 1024 addressable slots with 784 used.

Address Width Auto-Derivation

parameter addressWidth = $clog2(numWeight);
// $clog2(784) = 10 → 10-bit write address (covers 0–1023)

reg [addressWidth-1:0] w_addr; // 10-bit — write address
reg [addressWidth  :0] r_addr; // 11-bit — one bit wider intentionally!

r_addr is deliberately one bit wider than w_addr. It needs to count up to exactly numWeight (784) to trigger the "all weights consumed" condition. A 10-bit counter can only reach 1023 before wrapping — but the check r_addr == numWeight needs to fire cleanly at 784 without ambiguity.

Double-Width MAC Registers

reg [2*dataWidth-1:0] mul;  // 32-bit product  — prevents overflow
reg [2*dataWidth-1:0] sum;  // 32-bit accumulator
reg [2*dataWidth-1:0] bias; // 32-bit bias (value in upper 16 bits)

Multiplying two 16-bit values can produce a 32-bit result. Keeping the accumulator and bias at full 32-bit width preserves precision throughout the entire MAC chain — truncation to 16-bit only happens at the activation function output, not mid-computation.

Weight Loading — The Address Wrap Trick

always @(posedge clk) begin
    if(rst) begin
        w_addr <= {addressWidth{1'b1}}; // reset to 0x3FF (all ones)
        wen    <= 0;
    end
    else if(weightValid
            & (config_layer_num == layerNo)
            & (config_neuron_num == neuronNo)) begin
        w_in   <= weightValue;
        w_addr <= w_addr + 1; // first write: 0x3FF+1 = 0x000 (wraps to 0)
        wen    <= 1;
    end
    else
        wen <= 0;
end

On reset, w_addr is initialized to all-1s (0x3FF). The first incoming weight causes w_addr + 1 to wrap to 0x000 — so weight[0] lands at address 0 without needing a separate initialization state machine. The double filter (config_layer_num == layerNo AND config_neuron_num == neuronNo) ensures only this specific neuron responds to any given weight write on the shared bus.

Pretrained vs. Online Bias — Compile-Time Switch

`ifdef pretrained
    initial begin
        $readmemb(biasFile, biasReg); // load .mem binary at simulation/boot
    end
    always @(posedge clk) begin
        bias <= {biasReg[addr][dataWidth-1:0], {dataWidth{1'b0}}};
        // bias stored in UPPER 16 bits → aligns with 32-bit sum format
    end
`else
    always @(posedge clk) begin
        if(biasValid & (config_layer_num==layerNo) & (config_neuron_num==neuronNo))
            bias <= {biasValue[dataWidth-1:0], {dataWidth{1'b0}}};
    end
`endif

The `ifdef pretrained directive is a compile-time switch — only one branch ever exists in the synthesized netlist. In pretrained mode, bias values are read from a binary .mem file generated by the Python offline training script. In online mode, the ARM streams bias values over AXI at runtime. In both cases, the bias value is placed in the upper 16 bits of the 32-bit register ({value, 16'b0}) so it aligns correctly with the fixed-point position of the accumulated 32-bit sum.

Pipeline Timing — The Delay Chain

// Every signal is shifted by 1 cycle to match physical propagation delays
always @(posedge clk) begin
    myinputd     <= myinput;          // D+1: wait for weight memory read
    weight_valid <= myinputValid;     // D+1: w_out valid one cycle after r_addr steps
    mult_valid   <= weight_valid;     // D+2: product valid one cycle after operands
    muxValid_d   <= mux_valid;        // D+1: delayed copy for edge detect
    muxValid_f   <= !mux_valid & muxValid_d; // falling edge: last product done
    sigValid     <= ((r_addr == numWeight) & muxValid_f) ? 1 : 0;
    outvalid     <= sigValid;         // D+1: one clean output pulse
end
Cycle: 0 1 2 3 myinput [valid] · · · r_addr → N+1 → N+2 · · (steps with myinputValid) w_out · [valid] · · (memory takes 1 cycle) myinputd · [valid] · · (delayed to match w_out) mul · · [valid] · (product: both operands ready) mult_valid · · [valid] · sum updated · · · [valid] (accumulator updates)

Without myinputd, the multiplier would see the new input data one cycle before the weight memory has had time to output the corresponding weight. The delay chain keeps every signal in lockstep with the actual hardware latencies.

Saturating Accumulator — Overflow Prevention

always @(posedge clk) begin
    if(rst | outvalid)
        sum <= 0; // reset after each output

    else if((r_addr == numWeight) & muxValid_f) // last product done → add bias once
    begin
        if(!bias[MSB] & !sum[MSB] & BiasAdd[MSB])   // +ve + +ve = -ve → overflow
            sum <= MAX_POSITIVE;                      // clamp to 0111...1
        else if(bias[MSB] & sum[MSB] & !BiasAdd[MSB]) // -ve + -ve = +ve → overflow
            sum <= MIN_NEGATIVE;                      // clamp to 1000...0
        else
            sum <= BiasAdd;
    end

    else if(mux_valid) // accumulate each product with same saturation logic
    begin
        if(!mul[MSB] & !sum[MSB] & comboAdd[MSB])
            sum <= MAX_POSITIVE;
        else if(mul[MSB] & sum[MSB] & !comboAdd[MSB])
            sum <= MIN_NEGATIVE;
        else
            sum <= comboAdd;
    end
end

Signed overflow occurs when two numbers of the same sign produce a result with the opposite sign — e.g. 0101 + 0110 = 1011 (5 + 6 = −5 in two's complement). The three conditions check the MSBs of both operands and the result: if two positives produce a negative (or vice versa), the accumulator clamps to the maximum (or minimum) representable value instead of wrapping around. The bias is added exactly once via the muxValid_f falling-edge pulse — not on every cycle.

Activation Function — Synthesis-Time Generate Block

generate
    if(actType == "sigmoid") begin : siginst
        Sig_ROM #(.inWidth(sigmoidSize), .dataWidth(dataWidth)) s1(
            .clk(clk),
            .x(sum[2*dataWidth-1 -: sigmoidSize]), // top 5 MSBs of sum
            .out(out)
        );
    end
    else begin : ReLUinst
        ReLU #(.dataWidth(dataWidth), .weightIntWidth(weightIntWidth)) s1(
            .clk(clk),
            .x(sum),
            .out(out)
        );
    end
endgenerate

The generate block is eliminated at synthesis — only one branch is physically instantiated. For Sigmoid, sum[31 -: 5] extracts the top 5 bits as the LUT address (these MSBs capture the most significant part of the result). For ReLU, the full 32-bit sum is passed in and the weightIntWidth parameter tells the module where the integer/fractional boundary lies.

Module 2 — axi_lite_wrapper.v

This module is the complete ARM↔FPGA interface. The ARM Cortex-A9 communicates with the FPGA logic by reading and writing memory-mapped registers over the AXI4-Lite bus. This module is the AXI slave that receives those register accesses and translates them into control signals the neural network understands — weights, biases, layer configs, soft resets, and result readback.

The Register Map

// 8 × 32-bit registers mapped to the ARM's address space
reg [31:0] weightReg;   // addr 0x00 — write a weight value here
reg [31:0] biasReg;     // addr 0x04 — write a bias value here
reg [31:0] outputReg;   // addr 0x08 — READ the NN result from here
reg [31:0] layerReg;    // addr 0x0C — which layer are you loading?
reg [31:0] neuronReg;   // addr 0x10 — which neuron are you loading?
reg [31:0] statReg;     // addr 0x14 — status: 1=result ready, read-to-clear
reg [31:0] controlReg;  // addr 0x18 — bit[0]=softReset
reg [31:0] slv_reg7;    // addr 0x1C — read external NN data via axi_rd_en

These 8 registers are the complete API between the ARM software and the FPGA neural network. The address decoding uses bits [4:2] of the AXI address (stripping the bottom 2 byte-offset bits since all registers are 32-bit aligned):

axi_awaddr[4:2] → register index 0–7
ADDR_LSB = 2   (strip 2 byte-offset bits from 32-bit aligned bus)

ARM Software Workflow — What the C Code Does

// Pseudocode — what the bare-metal C running on ARM Cortex-A9 does

// Step 1: Tell the FPGA which neuron we're loading
write(layerReg,  0);           // "I'm loading layer 0"
write(neuronReg, 5);           // "I'm loading neuron 5"

// Step 2: Stream all weights — each write pulses weightValid for 1 cycle
for(int i = 0; i < 784; i++)
    write(weightReg, weights[i]);

// Step 3: Write the bias
write(biasReg, bias_val);

// Step 4: Run inference — send input vector
// (handled via DMA / AXI-Stream, not this lite interface)

// Step 5: Poll for result
while(read(statReg) == 0);     // spin until NN sets statReg=1
result = read(outputReg);        // grab the result — statReg auto-clears

AXI4-Lite Write Handshake — 3 Channels, 5 Signals

A single AXI4-Lite write requires three independent channels to complete in sequence:

AW Channel — Write Address

AWVALID / AWREADY handshake. The slave asserts AWREADY for one cycle when it is ready to accept an address. The address is latched into axi_awaddr at that cycle and held until the write completes.

W Channel — Write Data

WVALID / WREADY handshake. Pulses simultaneously with the AW handshake. The actual register write fires when axi_wready && WVALID && axi_awready && AWVALID are all high.

// The aw_en guard — prevents accepting new writes before response acknowledged
if(~axi_awready && S_AXI_AWVALID && S_AXI_WVALID && aw_en) begin
    axi_awready <= 1;   // pulse AWREADY — handshake accepted
    aw_en       <= 0;   // lock: no new writes until B channel clears
end
else if(S_AXI_BREADY && axi_bvalid) begin
    aw_en       <= 1;   // B channel acknowledged → unlock
    axi_awready <= 0;
end

aw_en is a critical guard flag. Without it, a fast ARM could issue back-to-back writes before the response for the previous write has been consumed, corrupting axi_awaddr. The flag enforces strict ordering: accept address → write data → send response → acknowledge response → unlock.

weightValid / biasValid — Single-Cycle Pulse Generation

always @(posedge clk) begin
    weightValid <= 1'b0;    // DEFAULT: deassert every single cycle
    biasValid   <= 1'b0;    // DEFAULT: deassert every single cycle

    if(slv_reg_wren) begin
        case(axi_awaddr[4:2])
            3'h0: begin
                weightReg   <= S_AXI_WDATA;
                weightValid <= 1;  // ← exactly 1 cycle pulse
            end
            3'h1: begin
                biasReg   <= S_AXI_WDATA;
                biasValid <= 1;    // ← exactly 1 cycle pulse
            end
            3'h3: layerReg   <= S_AXI_WDATA;
            3'h4: neuronReg  <= S_AXI_WDATA;
            3'h7: controlReg <= S_AXI_WDATA;
        endcase
    end
end

The weightValid <= 0 at the top of the always block is the key pattern. Every clock cycle, it is driven to 0 first. It only becomes 1 on the exact cycle that slv_reg_wren fires and the address matches register 0. This guarantees a guaranteed single-cycle pulse — exactly the one-clock-wide strobe that the neuron's weight loading block expects to step its write address and latch one value into BRAM.

Status Register — Read-to-Clear Flag

always @(posedge clk) begin
    if(reset)
        statReg <= 0;
    else if(nnOut_valid)
        statReg <= 1;         // NN finished — set the flag
    else if(axi_rvalid && S_AXI_RREADY
            && axi_araddr[4:2] == 3'h6) // ARM read statReg
        statReg <= 0;         // auto-clear on read
end

This implements a read-to-clear interrupt flag in hardware. When the neural network finishes a forward pass (nnOut_valid=1), statReg is set to 1. When the ARM reads address 0x14, the hardware automatically clears it — no separate write to clear needed. The ARM polling loop is just:

while(read(0x14) == 0);   // wait for result
result = read(0x08);   // outputReg — statReg auto-cleared

Soft Reset

assign softReset = controlReg[0];
// ARM writes 0x00000001 → asserts reset across entire NN
// ARM writes 0x00000000 → deasserts reset
// Level-sensitive: held high as long as bit 0 stays set

A single bit in the control register drives the global reset line to every neuron module. The ARM can generate a reset pulse by writing 1 then immediately writing 0 in consecutive AXI transactions — both completing in a handful of clock cycles.
